## 1. 정규화 (Normalization)
정규화는 데이터 중복을 최소화하고, 삽입·삭제·갱신 이상(Anomaly)을 방지하기 위해 릴레이션을 분해해나가는 과정이다. 각 단계는 이전 단계를 만족한 상태에서 추가 조건을 하나씩 더 만족시키는 방식으로 진행된다.

| 단계 | 조건 |
|---|---|
| **1NF** (제1정규형) | 모든 속성 값이 **원자값(더 이상 쪼갤 수 없는 단일 값)**이어야 함 → 다중값 속성 제거 |
| **2NF** | 1NF 만족 + **부분 함수 종속 제거** → 완전 함수 종속만 남김 |
| **3NF** | 2NF 만족 + **이행 함수 종속 제거** |
| **BCNF** | 3NF 만족 + **모든 결정자가 후보키(PK)** 여야 함 |
| **4NF** | BCNF 만족 + **다치 종속 제거** |
| **5NF** | 4NF 만족 + **조인 종속** 제거 |

### 1NF 예시

한 학생이 여러 과목을 수강해서 하나의 속성에 여러 값이 들어가 있는 경우(다중값 속성), 이를 원자값으로 쪼개서 행을 분리하는 예시야.
 
**수정 전** (수강과목 속성에 여러 값이 들어감 → 원자값 아님)

| 이름 | 수강과목 |
|---|---|
| 허민엽 | a, b |
 
**1NF 적용 후** (원자값으로 분리, 행을 나눔)

| 이름 | 수강과목 |
|---|---|
| 허민엽 | a |
| 허민엽 | b |
 
이렇게 하나의 셀에 여러 값이 들어가 있으면 검색·수정·삭제가 전부 까다로워진다. 예를 들어 권민영이 α 과목만 취소하고 싶어도, 문자열을 파싱해서 "α, b"에서 α만 골라내는 로직이 필요해진다. 행을 분리해두면 그냥 `WHERE 이름='권민영' AND 수강과목='α'` 조건으로 해당 행 하나만 삭제하면 끝난다.

### 2NF 예시 — 부분 함수 종속 제거

기본키가 두 개 이상의 속성으로 구성된 **복합키**일 때만 발생하는 문제야. 기본키의 일부(전체가 아닌 부분)에만 종속되는 속성이 있으면 2NF 위반이다.

**수강** 릴레이션 (기본키: 학번 + 과목번호)

| 학번 | 과목번호 | 학생이름 | 담당교수 |
|---|---|---|---|
| 001 | C001 | 김민엽 | 이교수 |
| 001 | C002 | 김민엽 | 박교수 |
| 002 | C001 | 허민엽 | 이교수 |

- `학번+과목번호` → `담당교수` : 두 속성 모두 있어야 결정됨 (완전 함수 종속, 문제 없음)
- `학번` → `학생이름` : 기본키 중 **학번 하나만으로** 결정됨 (**부분 함수 종속**, 2NF 위반)
**2NF 적용 → 분해**

`학생` 테이블

| 학번 | 학생이름 |
|---|---|
| 001 | 김민엽 |
| 002 | 허서연 |

`수강` 테이블

| 학번 | 과목번호 | 담당교수 |
|---|---|---|
| 001 | C001 | 이교수 |
| 001 | C002 | 박교수 |
| 002 | C001 | 이교수 |

이렇게 분리하면 김민엽이 이름을 바꿀 때 수강 테이블 여러 행을 다 고칠 필요 없이, 학생 테이블 한 행만 수정하면 된다 (갱신 이상 방지).

### 3NF 예시 — 이행 함수 종속 제거
 
기본키가 단일 속성이어도, 그 기본키가 결정하는 일반 속성이 또 다른 일반 속성을 결정하는 경우(A→B→C 형태의 사슬) 3NF 위반이야.
 
**사원** 릴레이션 (기본키: 사원번호)
 
| 사원번호 | 부서번호 | 부서위치 |
|---|---|---|
| E01 | D1 | 서울 |
| E02 | D1 | 서울 |
| E03 | D2 | 부산 |
 
- `사원번호` → `부서번호` (기본키가 직접 결정, 문제 없음)
- `부서번호` → `부서위치` (일반 속성이 또 다른 일반 속성을 결정)
- 따라서 `사원번호` → `부서번호` → `부서위치` (**이행 함수 종속**, 3NF 위반)
**3NF 적용 → 분해**

`사원` 테이블

| 사원번호 | 부서번호 |
|---|---|
| E01 | D1 |
| E02 | D1 |
| E03 | D2 |

`부서` 테이블

| 부서번호 | 부서위치 |
|---|---|
| D1 | 서울 |
| D2 | 부산 |

D1 부서가 이사 가면, 분해 전에는 사원이 몇 명이든 그 인원수만큼 행을 다 고쳐야 했지만, 분해 후에는 부서 테이블 한 행만 고치면 된다.

### BCNF 예시 — 결정자가 후보키가 아닌 경우

**일반 속성이 다른 속성을 결정하는 결정자인데 그 결정자가 후보키가 아닌 경우** 

**수강** 릴레이션 (한 학생은 여러 과목 수강 가능, 한 교수는 한 과목만 담당)

| 학생 | 과목 | 교수 |
|---|---|---|
| 김민엽 | DB | 이교수 |
| 김민엽 | OS | 박교수 |
| 허서연 | DB | 이교수 |

- 후보키: `(학생, 과목)`, `(학생, 교수)` — 둘 다 전체 행을 유일하게 식별 가능
- 그런데 `교수` → `과목` (한 교수는 한 과목만 가르치므로, 교수를 알면 과목이 결정됨)
- `교수`는 결정자이지만 후보키가 아님 → **BCNF 위반**
**BCNF 적용 → 분해**

`수강` 테이블
 
| 학생 | 교수 |
|---|---|
| 김민엽 | 이교수 |
| 김민엽 | 박교수 |
| 허서연 | 이교수 |

`개설` 테이블

| 교수 | 과목 |
|---|---|
| 이교수 | DB |
| 박교수 | OS |

### 4NF 예시 — 다치 종속 제거

한 개체가 서로 **독립적인 다중값 속성 두 개 이상**을 가질 때 발생하는 문제야.

**학생취미동아리** 릴레이션 (한 학생이 취미도 여러 개, 동아리도 여러 개, 서로 무관하게 조합됨)

| 학번 | 취미 | 동아리 |
|---|---|---|
| 001 | 독서 | 천체관측 |
| 001 | 독서 | 코딩 |
| 001 | 게임 | 천체관측 |
| 001 | 게임 | 코딩 |
 
취미가 2개, 동아리가 2개라서 **2×2=4개 행**이 강제로 만들어진다. 취미와 동아리는 서로 아무 관련이 없는데도 모든 조합을 다 나열해야 하는 게 다치 종속 문제(`학번 →→ 취미`, `학번 →→ 동아리`가 독립적으로 존재).
 
**4NF 적용 → 분해**
 
`학생취미`
 
| 학번 | 취미 |
|---|---|
| 001 | 독서 |
| 001 | 게임 |
 
`학생동아리`
 
| 학번 | 동아리 |
|---|---|
| 001 | 천체관측 |
| 001 | 코딩 |

## 2. 반정규화 (De-normalization)

- 정규화된 테이블의 규칙을 **의도적으로 통합·중복 허용**하여 **DB 성능을 향상**시키는 기법
- 정규화를 철저히 지킬수록 데이터 무결성은 좋아지지만, 테이블이 여러 개로 쪼개지면서 JOIN 연산이 많아져 조회 성능이 떨어질 수 있다. 반정규화는 이 트레이드오프에서 **조회 속도를 우선시**할 때 사용하는 기법
- 스마트팩토리 MES처럼 실시간 조회가 잦은 시스템(예: 설비 상태 대시보드, 생산 실적 집계)에서는 통계·집계용 테이블을 일부러 반정규화해서 매번 여러 테이블을 JOIN하지 않고 빠르게 조회하도록 설계하는 경우가 많음

## 3. 관계 대수 (Relational Algebra)

- **정의**: DB에서 원하는 데이터를 찾는 **절차형(How) 정형 언어**
- **관계 대수**: 절차적 언어 (어떻게 찾을지 순서를 명시)
- **관계 해석**: 비절차적 언어 (무엇을 찾을지만 명시)
### 관계 대수 연산자

**일반 집합 연산자**
 
| 연산자 | 기호 | 설명 |
|---|---|---|
| ① 합집합 | ∪ | 두 릴레이션의 합집합 |
| ② 교집합 | ∩ | 두 릴레이션의 교집합 |
| ③ 차집합 | − | 두 릴레이션의 차집합 |
| ④ 카티션 프로덕트(Cartesian Product) | × | 두 릴레이션의 **순서쌍 조합**(product) |

**순수 관계 연산자**

| 연산자 | 기호 | 설명 |
|---|---|---|
| ⑤ 셀렉트(Select) | σ | 조건을 만족하는 **행(튜플)**을 추출 |
| ⑥ 프로젝트(Project) | π | 한 개~한 집합의 **속성(컬럼)**만 추출 |
| ⑦ 조인(Join) | ⋈ | 공통 속성을 기준으로 두 릴레이션을 결합 |
| ⑧ 디비전(Division) | ÷ | 특정 조건을 만족하는 튜플 추출(나눗셈 개념) |

## 4. 데이터 무결성 (Data Integrity)

암기 포인트: **개참도사키**

| 무결성 종류 | 내용 |
|---|---|
| ① **개체 무결성** | PK를 사용하여 튜플을 구별. **NOT NULL**이고 릴레이션 내에서 **UNIQUE** → PK 조건 |
| ② **참조 무결성** | **외래키(FK)** 사용. FK 값은 참조하는 부모 릴레이션에 **실제 존재하는 값이거나 NULL**이어야 함 |
| ③ **도메인 무결성** | 속성 값이 **DOMAIN, CHECK, DEFAULT** 등으로 미리 정의된 데이터 타입·범위 내에 있어야 함 |
| ④ **사용자 정의 무결성** | 업무 규칙에 따라 **TRIGGER**를 사용해서 구현 |
| ⑤ **키 무결성** | **UNIQUE** 사용. 하나의 릴레이션에는 적어도 1개의 키가 존재해야 함 |

### 스마트팩토리 도메인 적용 예시

- **개체 무결성**: 설비 테이블에서 설비ID를 PK로 지정해 NULL·중복 없이 각 설비를 유일하게 식별
- **참조 무결성**: 생산실적 테이블의 설비ID(FK)는 반드시 설비 마스터 테이블에 존재하는 값이어야 함 → 존재하지 않는 설비의 생산실적이 입력되는 오류 방지
- **도메인 무결성**: 온도 센서 값 컬럼에 CHECK 제약조건으로 -50~200(℃) 범위를 벗어난 이상값 입력 자체를 차단
- **사용자 정의 무결성**: 특정 설비 상태가 "고장"으로 바뀌면 TRIGGER로 알림 테이블에 자동 기록
