## 소프트웨어 개발 방법론

### 컴퓨터 시스템의 기본 요소
* **입력(input)**: 데이터, 제어 요소 전달
* **처리(process)**: 방식에 맞게 처리하여 결과 도출
* **출력(output)**: 처리 결과 -> 출력 장치 및 저장 장치 전달
* **제어(control)**: 데이터 처리를 위해 기능 수행 제어
* **피드백(feedback)**: 기능 수행 잘못 -> 처리 과정 다시 반복

**정의**

3. sw 단계별 결과물 -> 명확한 기록

### sw 개발 프레임워크

**모듈**
* 분할하여 재사용 -> 부품화
* 기본 틀 제공

**라이브러리(툴킷)**
* 관련있는 모듈 모아둔 것
* **표준 라이브러리**: 프로그래밍 언어에 내장
* **외부 라이브러리**: 별도의 설치를 통해 사용 가능

**디자인 패턴**
* 특정 기능에 대한 문제해결 -> 추상적 가이드라인 제시

**sw 개발 프레임워크**
* 디자인 패턴 + 모듈 -> 실질적인 개발 틀
* 개발자: 제어의 역 흐름 발생
* 특정 디자인 패턴 유도
* 품질, 예산, 유지보수이 장점

**컴포넌트**
* 모듈 형태, 재사용 가능, 확장된 sw 블럭
* 일반적 코딩 x, 레고(블럭)처럼 컴포넌트 결합
* 인터페이스 제공
* **협약에 의한 설계**
    1. 선행조건: 컴포넌트 오퍼레이션 사용 전에 참이어야 하는 조건
    2. 결과조건: 컴포넌트 오퍼레이션 사용 후에 참이어야 하는 조건
    3. 불변조건: 컴포넌트 오퍼레이션 실행 중에 참이어야 하는 조건

**재사용 가능한 sw 요소**
* 모두 재사용 요소로 가능 (데이터 구조, 테스트 계획, 문서화 방법, 절차 등...)
* **sw 재사용 방법**
    1. 합성 중심: 모듈(컴포넌트) 조립 -> sw 완성
    2. 생성 중심: 추상적인 명세 -> 구체화 -> sw 완성

### 소프트웨어 개발 수명 주기

#### 소프트웨어 모델
* **폭포수 모델**
    * 정해진 단계 한번씩만 진행, 이전 단계 되돌아 x
    * 명확한 산출물이 있어야 -> 다음 단계
    * 보완 불가능 -> 메뉴얼 작성 필수
* **프로토타입 모델**
    * 폭포수 모델 단점 보안(이전 단계로 못간다는 단점)
    * 시제품(프로토타입)을 통해 최종 결과물 예측
    * 사용자와 시스템 사이 인터페이스 중심
* **나선형 모델**
    * 폭포수 모델 + 프로토타입 + 위험 분석 기능
    * 점진적 개발
    * 유지보수 필요 x
* **애자일 모델**
    * sw 사용 고객 소통 중심 방법론의 총칭
    * 짧은 개발 주기 반복 -> 고객과 피드백 반영
    * 개발 모델: Scrum, XP, Kanban, crystal, FDD, ASD, DSDM
    * 가치: 소통, 협업, 변화 대응

---

#### Scrum
**특징**
1. 스크럼 팀 구성: 제품 책임자, 스크럼 마스터, 개발팀
2. 반복적 스프린트 -> 제품 완성
3. 스프린트: 하나의 task를 개발과정(2~4주)
4. 가치: 확약, 전념, 정직, 존중, 용기

**제품 책임자**
* 제품에 대한 모든 책임, 의사 결정 역할
* 백로그 작성
    * 제품 백로그: 우선 순위에 따라 개발에 필요한 사용자 스토리
    * 스프린트 백로그: 스프린트에서 개발해야할 task 나열 목록
    * 사용자 스토리: 사용자 요구사항 -> 단어 나열 x, 스토리 o
    * 릴리즈 계획: 전체 개발 일정 수립
* 팀원들은 스토리만 추가 o, 우선순위 조정 x

**스크럼 마스터**
* 개발팀원 원활 업무를 위한 가이드 역할
* 일일 스크럼 회의 -> 발생한 장애 요소 해결
* 상황에 유연하게 대응 o, 통제 권한 x

**개발팀**
* 모든 개발에 참여하는 인원 (로그인 개발팀, ux 개발팀 ...)

---

#### XP (eXtreme Programming)
**특징**
* 짧은 개발 과정 반복 극대화 -> 개발 생산성 높임
* 소규모 인원
* 가치: 의사소통, 단순성, 용기, 존중, 피드백

**개발 프로세스**
1. 소규모 릴리즈
2. 스파이크: 특정 기술 확인 -> 간단하게 개발하는 프로그램
3. 이터레이션
    * 스파이크 기술 검증 -> 이터레이션 전달
    * 이터레이션 도중 -> 새로운 스토리 추가 o

**XP 실천 사항**
1. planning game
2. small releases
3. system metaphor
4. simple design
5. test driven development
6. design improvement
7. pair programming
8. collective ownership
9. continuous integration
10. sustainable pace
11. whole team
12. coding standards

### 소프트웨어 개발 방법론
* **정의**: sw 개발 전 과정 지속적 적용할 수 있는 방법, 절차, 기법
* **목적**: sw 개발에 있어 생산성, 품질 향상 = 효율을 올리기 위함
* **절차**: 분석(개발 준비, 시스템/sw 요구사항 분석) -> 설계(시스템 설계, sw 구조 및 상세 설계) -> 구현(sw 코딩 및 단위 시험) -> 시험(sw와 시스템 통합, 테스트)

**방법론 종류**
1. **구조적 방법론**
    * yourdon이 개발
    * 자료 흐름도(DFD): 프로그램 기능 단위별 데이터 흐름
    * 자료 사전(DD): DFD에 표현된 자료 저장소 구체화
    * 모듈 중심 설계 -> 독립성 높임
    * 순차, 선택, 반복 -> 복잡성 최소
2. **정보공학 방법론**
    * 개발 단계별 정형화된 기법 -> 통합 적용 -> 데이터 중심 방법론
    * 데이터 모델링 도구: 개체-관계 다이어그램(ERD)
    * 프로세스 모델링 도구: 자료 흐름도, 프로세스 의존도(PDD), 프로세스 계층도(PHD)
3. **객체지향 방법론**
    * 실체 -> 객체 (속성: 성질/값/데이터, 메소드: 동작)
    * 객체 저장 -> 관계형 테이블로 변환하는 과정 필요
    * 기본 원칙: 캡슐화, 정보은닉, 추상화, 상속성, 다형성
4. **컴포넌트 기반(CBD)**
    * 컴포넌트 조립 방식
    * 장점: 개발 생산성, 이식성, 호환성, 신속성, 유연성, 표준화
    * 단점: 선행 투자 비용 높음, 책임 및 지적 재산권 고려 필요
    * 단계별 산출물: 분석(정의서, 명세서), 설계(설계서, 계획서), 구현/시험(결과서)
5. **애자일 방법론**
    * 위 내용 참조
6. **제품 계열 방법론**
    * 임베디드 sw 작성때 유용
    1. 영역공학: 영역 분석, 영역 설계, 핵심 자산 구현
    2. 응용공학: 제품 요구분석, 제품 설계, 제품 구현

### 소프트웨어 보안 개발 방법론
* **정의**: sw 보안 취약점 최소화 -> 지침 및 사례 기반
* **종류**
    1. **MS-SDL**: MS에서 개발한 sw 보안 개발 방법론
    2. **Seven Touchpoints**: 7가지 모델 통합 (코드 검토, 아키텍처 위험 분석, 침투 테스트, 위험 기반 보안 테스트, 악용 사례, 보안 요구, 보안 운영)
    3. **CLASP**: 이미 운영중인 시스템에 적용 용이 (5가지 관점: 개념, 역할 기반, 활동 평가, 활동 구현, 취약성)
    4. **CWE**: sw 보안 취약점 -> 체계적 분류한 DB
