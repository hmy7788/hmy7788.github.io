---

### 2) 유스케이스 다이어그램
#### 유스케이스 다이어그램 정의
* 수행하는 기능을 사용자 관점에서 표현한 도표
* 시스템 범위 파악, 시스템 간 상호 작용 파악 가능

#### 유스케이스 다이어그램 구성 요소
* 시스템 범위: 유스케이스를 사각형
* 사용자(주) 액터: 상호작용 -> 이득을 얻는 대상
* 시스템(부) 액터: 외부 시스템
* 유스케이스: 사용자 관점의 제공 서비스, 기능을 단순하게 표현
* 관계: 유스케이스 -> 유스케이스, 액터 -> 유스케이스 관계 표현

#### 유스케이스 작성 시 주의사항
* 실행 순서 나타냄 x -> 흐름도처럼 작성 x
* 최대한 간결하게 표현

---

### 3) 유스케이스 다이어그램 관계
#### 포함 관계(필수적)
* 공통으로 사용되는 기능 추출 -> 새로운 유스케이스 생성 & 연결
* 기존 유스케이스 -> 새로운 유스케이스 방향, 점선 화살표 연결 & <\<include>> 표기

#### 일반화 관계
* 유스케이스 간 상하 관계 표현
* 하위 유스케이스 -> 상위 유스케이스 방향, 속이 빈 실선 화살표 연결

#### 확장 관계(선택적)
* 특정 조건에서만 실행되는 유스케이스 생성 & 연결
* 선택적 유스케이스 -> 원래 유스케이스 방향, 점선 화상표 연결 & <\<extend>> 표기

## 3. 요구사항 분석
### 1) 요구사항 분석
#### 요구사항 분석 정의
* 요구사항을 조정해 나가는 활동
* 명확하지 않거나, 상충되는 요구사항 -> 분석 & 문제점 해결

---

### 2) 요구사항 분석 기법
#### 요구사항 분류
* 소프트웨어 관점 요구사항: 역량, 성능, 기능
* 기능적 요구사항: 시스템 기능, 제어 연산, 기술에 대한 요구사항
* 비기능적 요구사항: 성능, 보안, 품질, 안정에 대한 요구사항
* 사용자 요구사항: 사용자에게 제공해야 하는 것, 친숙한 표현
* 시스템 요구사항: 시스템에게 제공해야 하는 것, 기술적 표현
* 요구 사항 출처, 우선순위, 적용 범위, 변경 가능성 식별

#### 개념 모델링
* 업무 처리의 실체, 관게를 모델링
* 요구사항 분석의 핵심 단계
* UML

#### 요구사항 할당
* 요구사항을 만족시키기 위한 구성요소 식별

#### 요구사항 협상
* 요구사항 상충 -> 협의하는 과정
* 하나를 포기하기보다는 우선순위 부여 -> 적절한 기준점 찾기(합의)
  * 두 명의 이해관계자가 요구하는 요구사항이 서로 충돌하는 경우
  * 요구사항과 자원이 서로 충돌하는 경우
  * 기능적 요구사항과 비기능적 요구사항이 서로 충돌하는 경우

### 3) 구조적 분석
#### 구조적 분석의 특징
* 도형 중심으로 표현 도표
* 구조적 분석 기본 원리 4가지: 추상화 원칙, 정형화 원칙, 분항 정복, 계층화

#### 자료 흐름도
* 데이터의 흐름을 도형으로 표현한 도표
* 제어 흐름 X, 데이터 흐름 중심 O, 작업 소요 시간 파악 불가능

| 구성 | 설명 |
| :--- | :--- |
| 프로세스 | 데이터 처리 과정, 타원 표기 |
| 자료 흐름 | 데이터 흐름 방향, 화살표 표기 |
| 자료 저장소 | 데이터 저장 공간, 상하 평행선 표기 |
| 단말 | 데이터 입출력 주체, 사각형 표기 |

#### 자료 사전
* 자료 흐름도에 사용된 데이터의 이름과 속성을 표기한 자료 (=메타 데이터)

| 기호 | 규칙 | 예시 |
| :--- | :--- | :--- |
| 정의 | = 표기, 자료 정의 | 대여번호 = 대여일 + 회원번호 + 일련번호 |
| 연결 | + 표기, 두 데이터 연결 | (위 예시의 '+' 기호 참조) |
| 선택 | [], \| 표기, 복수 데이터 중 선택 | 회원등급 = ["G" \| "S" \| "B"]|
| 반복 | {} 표기, 반복되는 데이터 | 회원별 대여 기록 = {회원번호} + 일련번호 |
| 생략 | () 표기, 생략 가능 데이터 | 주소 = (회원의 거주지 주소) |
| 주석 | *으로 감싸서 표기, 데이터 설명 | 회원등급 = \*G: 골드, S: 실버, B: 브론즈\* |

#### NS 차트
* 논리 중심 표현 차트
* 순차, 선택, 반복 제어구조 명확히 표현
* 임의 제어 이동 어려움

#### HIPO
* 기능, 데이터 관계 -> 모듈화 분할 -> 계층 구조
* 기능, 데이터 의존성 동시 표현 -> 하향식 소프트웨어 개발에 용이

| 도표 | 기능 |
| :--- | :--- |
| 가시적 도표 | 전체적 기능, 흐름 -> 트리 형태 구조도 |
| 총체적 도표 | 입력, 처리, 출력 -> 전반적 정보 제공 도표 |
| 세부적 도표 | 총체적 도표 기본 요소 -> 상세히 기술 도표 |
