## 1. 키(Key)
 
### 키의 성질
- **유일성(Uniqueness)**: 하나의 키 값으로 튜플(행)을 유일하게 식별할 수 있는 성질
- **최소성(Minimality)**: 유일성을 만족하는 속성 중, 꼭 필요한 최소한의 속성으로만 구성되는 성질

### 기본키 (PK, Primary Key)
- 후보키 중 **선택된 단 하나의 키**. 테이블의 대표 키 역할을 함
- 조건
  - NULL 값 불가
  - 중복된 값 불가
  - 테이블 당 1개의 PK만 지정 가능
- 예시: 학번, 주민번호

### 대체키 (Alternate Key)
- 후보키 중 **PK로 선택되지 않은 나머지 키들**
- 유일성 O, 최소성 O
- 예시: 학번이 PK로 선택되었다면, 이메일은 대체키가 됨

### 후보키 (Candidate Key)
- 테이블의 각 행을 유일하게 식별할 수 있는 **최소한의 속성(또는 속성 집합)**
- 유일성 O, 최소성 O
- 예시
  - 학번 → 후보키 O
  - 이메일 → 후보키 O
  - 이름 → 후보키 X (유일성 불만족)
  - (학번, 이메일) 조합 → 후보키 X (최소성 불만족, 학번만으로 이미 식별 가능)

### 슈퍼키 (Super Key)
- 테이블의 각 행을 유일하게 식별하는 속성 또는 속성 집합
- 유일성 O, 최소성 X (불필요하게 많은 속성이 포함되어도 됨)
- 예시: (학번, 이메일) → 슈퍼키 O

### 외래키 (FK, Foreign Key)
- 다른 테이블의 PK를 참조하는 속성

### 키 관계 정리
```
[기본키(PK) + 대체키] = 후보키(CK)
후보키(CK) ⊂ 슈퍼키(SK)
 
즉, PK ⊂ CK ⊂ SK
```
 
- Foreign Key는 위 포함관계와 별개로, 테이블 간 참조 관계를 나타내는 키

## 2. 릴레이션(Relation) 구성요소
릴레이션 = 2차원 테이블
| 구성요소 | 설명 |
|---|---|
| 튜플(Tuple) | 테이블의 각 행(Row). 레코드라고도 하며, 중복 가능 |
| 속성(Attribute) | 테이블의 각 열(Column). 필드라고도 함 |
| 카디널리티(Cardinality) | 튜플의 개수 |
| 차수(Degree) | 속성의 개수 |
| 스키마(Schema) | DB의 구조와 제약조건에 대한 명세 |
| 인스턴스(Instance) | 실제 저장된 데이터의 집합 |
| 도메인(Domain) | 하나의 속성이 가질 수 있는 모든 원자값들의 범위 |
 
### 예시 테이블
| 학번 | 이름 | 수강과목 | 나이 |
|---|---|---|---|
| 0 | 김 | DB | 20 |
| 1 | 박 | OS | 21 |
| 2 | 이 | OS | 21 |
| 3 | 허 | 알고리즘 | 24 |
| 4 | 악 | 네트워크 | 26 |
 
- 튜플 = 각 행 (총 5개)
- 속성 = 학번, 이름, 수강과목, 나이 (총 4개)
- **카디널리티 = 5**
- **차수 = 4**

## 3. 데이터 모델의 3가지 구성요소
1. **구조(Structure)**: 데이터의 논리적 구조
2. **연산(Operation)**: 데이터를 처리하는 방법
3. **제약조건(Constraint)**: 데이터의 무결성을 보장하는 규칙
---

## 4. DB 설계 4단계
```
요구사항 분석 → 개념적 설계 → 논리적 설계 → 물리적 설계 → 구현
```
| 단계 | 산출물 | 설명 |
|---|---|---|
| ① 요구사항 분석 | 요구사항 분석서 | 사용자가 필요로 하는 데이터와 처리 요건을 수집·정리 |
| ② 개념적 설계 | E-R 다이어그램 | 현실 세계를 개체(Entity)와 관계(Relationship) 중심으로 추상화. 특정 DBMS에 종속되지 않는 개념적 모델 |
| ③ 논리적 설계 | 논리적 스키마, 테이블 명세서 | 개념적 설계 결과를 특정 DBMS(관계형/계층형/네트워크형 등)가 이해할 수 있는 데이터 모델로 변환. 정규화 수행 |
| ④ 물리적 설계 | 물리적 스키마, 테이블 정의서 | 논리적 스키마를 실제 저장 구조(파일 구조, 인덱스, 접근 경로 등)로 구체화 |
| ⑤ 구현 | 실제 데이터베이스 | DDL로 테이블 생성, DML로 데이터 적재 등 물리적 설계를 실제 DBMS에 구현 |

> **정규화(Normalization)**는 논리적 설계 단계에서 수행되며, 데이터 중복을 최소화하고 삽입·삭제·갱신 이상(Anomaly)을 방지하기 위한 과정이다.

## 5. E-R 다이어그램 표기법

| 기호 | 의미 |
|---|---|
| 사각형 (□) | 개체 집합(Entity Set) |
| 마름모 (◇) | 관계 집합(Relationship Set) |
| 타원 (O) | 속성(Attribute) |
| 이중 타원 | 다중값 속성 |
| 밑줄 친 타원 | 기본키 속성 |
| 실선 (—) | 개체(Entity) ↔ 관계(Relationship) 연결 |
| 점선 (- - -) | 관계(Relationship) ↔ 속성(Attribute) 연결 |

### 정리하면

E-R 다이어그램은 크게 세 종류의 도형(사각형·마름모·타원)과 두 종류의 선(실선·점선)으로 구성된다.

- **사각형(개체)**과 **마름모(관계)** 사이는 **실선**으로 연결한다. 즉, "어떤 개체가 어떤 관계에 참여하는가"를 나타낸다.
  - 예: `학생 — 수강 — 과목` (학생 개체와 과목 개체가 수강이라는 관계로 실선 연결)
- **마름모(관계)**와 **타원(속성)** 사이는 **점선**으로 연결한다. 즉, "그 관계 자체가 가지는 속성"을 나타낸다.
  - 예: 수강이라는 관계 자체에 "수강일자", "성적" 같은 속성이 있다면, 이는 마름모에서 점선으로 뻗어나간 타원으로 표현
- 참고로 개체가 가지는 일반 속성(예: 학생의 이름, 학번)은 사각형(개체)에서 **실선**으로 타원과 연결하는 경우가 일반적이며, 여기서 말하는 점선은 특별히 "관계 자체의 속성"을 표시할 때 쓰인다는 점에서 구분된다.
